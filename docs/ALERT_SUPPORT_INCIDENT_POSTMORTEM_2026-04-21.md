# 알림/지원금 이슈 회고 (2026-04-21)

## 목적
- 같은 실수를 반복하지 않기 위한 내부 참고 문서.
- 알림 누락/불일치, 지원금 합산 불일치, 사용자 식별 혼선 이슈의 원인과 재발 방지 기준 정리.

## 사용자가 요구한 최종 기준
1. 관리자 `조별추첨확정현황`에 표시된 지원금 원본값이 그대로 지급 반영되어야 한다.
2. 같은 원본값이 `쇼핑지원금이 지급되었습니다` 알림 메시지에도 그대로 표시되어야 한다.
3. 마이페이지 알림 카운트와 알림 리스트가 같은 기준으로 보여야 한다.
4. 이전 알림(읽음 포함)도 페이지 뒤에서 조회 가능해야 한다.

## 실제 문제 원인(핵심)
1. **사용자 식별자 혼선**
   - `userId`, Firebase `uid(문서ID)`, `memberId`를 경로마다 다르게 사용.
   - 어떤 경로는 알림 저장 성공, 어떤 경로는 조회 실패로 누락처럼 보임.

2. **비원자적 지급 처리**
   - 일부 흐름에서 `paymentStatus=paid`를 먼저 반영하고, 그 뒤 토큰/알림 처리.
   - 중간 실패 시 상태만 paid, 실제 토큰 반영/알림은 누락 가능.

3. **알림 조회 기준 불일치**
   - 카운트는 unread 기준, 리스트는 전체 또는 다른 키 기준으로 조회.
   - 결과적으로 카운트와 리스트 개수가 안 맞아 보임.

4. **중복방지 로직의 부작용**
   - `support_paid_<id>` 고정 ID 방식은 중복은 막지만, “매번 새 알림 누적” 기대와 충돌 가능.

## 재발 방지 원칙(필수)
1. **단일 지급 원천**
   - 지급 금액은 반드시 `lotteryConfirmedResults.support` 원본만 사용.
2. **원자적 처리**
   - `tokenBalance 반영 + paymentStatus 변경`은 트랜잭션 1회 처리.
3. **알림 식별 통일**
   - 저장/조회 모두 최종 `userId` 기준으로 통일.
4. **카운트/리스트 동일 기준**
   - unread/전체 모드 정의를 분리하되, UI에서 명확히 표시.
5. **실패 가시화**
   - 에러 무시 금지. 사용자/관리자에게 실패 건수 노출.

## 작업 시 체크리스트
- [ ] 지급 금액 source는 `support` 한 곳인가?
- [ ] paid 업데이트와 tokenBalance 업데이트가 트랜잭션인가?
- [ ] 알림 저장 user key와 조회 user key가 같은가?
- [ ] 알림 카운트와 리스트의 필터(읽음/전체)가 같은가?
- [ ] 실패를 `catch`로 삼키지 않고 로그/메시지로 노출하는가?
- [ ] 페이지네이션 기준(20개/페이지)이 카운트 표기와 일치하는가?

## 금지 사항
- 임시 방어 로직을 사용자 합의 없이 운영 로직에 넣지 말 것.
- 실패를 `catch`에서 조용히 무시하지 말 것.
- 식별자(`userId`, `uid`, `memberId`) 변환을 함수마다 제각각 구현하지 말 것.
