# 개발환경에서만 "상품 상세가 한 경로로만 열리는" 문제 — 해결 정리

> 메인/상품목록에서 아무 상품을 눌러도 항상 같은 상품(예: 패리스독 피너츠라운지베드 베이지) 상세로만 이동하던 현상의 원인과 해결 방법, 그리고 왜 해결이 오래 걸렸는지 정리한 문서입니다.

---

## 1. 최종 해결 방법

### 1.1 원인

- **배포(프로덕션)** 에서는 정상: 상품별로 다른 상세 페이지로 이동함.
- **개발환경(localhost)** 에서만: 모든 상품 클릭 시 같은 상품 상세로 이동하거나, "상품을 선택해 주세요"만 표시됨.

**근본 원인 두 가지:**

1. **브라우저 캐시**  
   로컬에서 `script.js`, `products-list.js` 등이 **예전 버전으로 캐시**되어, 수정한 코드(상품 ID를 링크에 넣는 로직)가 실행되지 않음.  
   → 목록에서 만들어지는 링크에 `?id=상품ID`가 안 붙거나, 잘못된 id가 붙음.

2. **URL 의존**  
   상세 페이지는 **주소의 `?id=...`만** 보고 Firestore에서 상품을 불러옴.  
   주소에 id가 없으면(캐시된 JS 때문에 링크가 `product-detail.html`만 있음) "상품을 선택해 주세요"만 보이거나, 예전에는 첫 번째 상품으로 폴백됨.

### 1.2 적용한 해결

**메인 페이지와 상세 페이지가 "같은 데이터 소스(Firestore)"를 쓰되, 메인에서 고른 상품을 URL 말고도 넘길 수 있게 함.**

| 위치 | 내용 |
|------|------|
| **메인 페이지 (script.js)** | 상품 카드 클릭 시 **선택한 상품의 문서 ID를 `sessionStorage.setItem('selectedProductId', id)` 로 저장**한 뒤 `product-detail.html?id=...` 로 이동. |
| **상세 페이지 (product-detail.js)** | 상품 ID를 **1) URL의 `?id=` 로 먼저 읽고, 2) 없으면 `sessionStorage.getItem('selectedProductId')` 로 읽음.** 둘 다 없을 때만 "상품을 선택해 주세요" 표시. Firestore는 기존처럼 `products` 컬렉션에서 해당 id로 조회. |
| **정리** | 로드 성공 시 `sessionStorage.removeItem('selectedProductId')` 로 정리. URL에 id가 없었으면 `history.replaceState` 로 주소창에 `?id=...` 반영. |

그래서 **캐시 때문에 링크에 id가 안 붙어도**, 메인에서 클릭할 때 저장해 둔 id를 상세 페이지가 읽어서, **같은 Firestore 데이터**로 해당 상품만 보여 주도록 했음.

---

## 2. 왜 해결이 오래 걸렸는지 (AI/개발 관점)

### 2.1 캐시를 “완전히” 제어할 수 없었음

- **시도한 것:**  
  - `script.js?v=33`, `?v=100`, `?v=101` 등 **쿼리 스트링 버전** 올리기  
  - localhost일 때만 **`script.js?v=Date.now()`** 로 매번 새 URL 부르기  
  - 상품 ID를 **배열/ data 속성/ 클릭 시 그리드에서 읽기** 등으로 링크와 분리
- **한계:**  
  - **index.html 자체가 캐시**되면, 예전 HTML이 로드되고 예전 `script.js` URL이 요청됨.  
  - **서버 Cache-Control** 은 `serve-dev.js` 또는 서버 설정 영역이라, 코드만으로 “HTML/JS 절대 캐시 안 함”을 보장하기 어려움.  
  - **사용자 브라우저 캐시**는 개발자가 코드로 “지워라”라고 할 수 없고, 강력 새로고침/사이트 데이터 삭제는 사용자가 해야 함.

그래서 “코드만 고치면 무조건 개발환경에서도 된다”라고 보장할 수 없었고, **URL에 id가 안 붙는 상황**이 계속될 수 있었음.

### 2.2 원인을 “URL + 캐시”로만 보았음

- 상세 페이지는 원래 **URL의 `?id=` 만** 보고 동작한다고 가정하고,  
  “목록 쪽에서 반드시 `product-detail.html?id=상품ID` 로 링크를 만들어야 한다”에만 초점을 맞춤.
- 그래서 **링크 생성 쪽(스크립트 캐시, doc.id 덮어쓰기, data 속성, 클릭 핸들러)** 만 계속 손댐.
- **“메인과 같은 데이터를 쓰면 되니까, URL 말고 다른 경로(예: sessionStorage)로 id만 넘기면 된다”** 는 쪽으로는 상대적으로 늦게 전환함.

즉, **“같은 데이터 소스(Firestore)를 쓰되, 전달 방식만 URL 이외로 보강한다”**는 해결을 더 일찍 적용했으면 더 빨리 끝났을 수 있음.

### 2.3 사용자 환경을 직접 볼 수 없음

- 실제로 **어떤 HTML/JS가 로드됐는지**, **강력 새로고침/캐시 삭제가 제대로 됐는지**는 제가 직접 확인할 수 없음.
- “캐시 비웠는데도 안 된다”는 말만으로는,  
  - index.html 캐시가 남아 있는지,  
  - 서비스 워커/다른 스크립트가 개입하는지  
  등을 코드만 보고는 단정하기 어려웠음.

그래서 **“캐시만 비우면 된다”**는 설명으로는 해결이 안 되는 경우를 대비한 **대안(URL 말고 sessionStorage로 id 전달)** 이 필요했음.

## 3. 깃 이전 코드 그대로 가져와도 개발환경에서만 안 됐던 이유

- **깃 이전 코드 = 배포 환경에서 정상 동작하던 코드**라고 가정함.
- 그 코드를 **로컬에 그대로** 가져와도, **개발환경(localhost)에서만** “한 경로로만 간다” / “상품을 선택해 주세요”만 나온 이유는 아래와 같음.

### 3.1 브라우저가 예전 JS/HTML을 캐시해서 씀

- **배포:**  
  배포할 때마다 새 파일이 올라가고, 사용자는 새 URL(또는 새 쿼리)로 리소스를 받음.  
  그래서 **최신 코드**가 실행됨.
- **로컬:**  
  같은 주소(`http://localhost:5000/`, `.../script.js` 등)를 반복해서 열기 때문에,  
  브라우저가 **예전에 받아 둔 HTML/JS**를 그대로 씀.
- 따라서 **디스크/깃에는 최신 코드가 있어도**,  
  **실행되는 건 예전에 캐시된 스크립트**일 수 있음.  
  → 목록에서 만드는 링크가 `product-detail.html` 만 있거나, id가 비어 있거나, 잘못된 id가 붙어서,  
  상세 페이지는 “id 없음” 또는 “같은 id”만 받게 됨.

### 3.2 “같은 코드”가 “같은 환경”이 아님

- **같은 코드**여도  
  - **배포:** 매번 새로 받는(또는 캐시가 덜 쌓인) 환경  
  - **로컬:** 오래 쌓인 캐시가 있는 환경  
  이라서, **실제로 실행되는 파일**이 다를 수 있음.
- 그래서 **깃 이전 코드를 그대로 가져와도**,  
  **로컬에서는 예전에 캐시된 버전이 돌아가면** 개발환경에서만 이상하게 동작함.

### 3.3 정리

- **“깃 이전 코드 = 되는 코드”** 인데도 **개발환경에서만 안 됐던 이유**는,  
  **실행되는 게 깃에 있는 최신 코드가 아니라, 브라우저에 캐시된 예전 HTML/JS** 이기 때문임.  
- 그래서 **최종 해결**은  
  “캐시에 상관없이, 메인에서 고른 상품 id를 **URL 말고도 sessionStorage로 넘기고**,  
  상세는 **URL + sessionStorage** 둘 다 보고 **같은 Firestore**에서 데이터를 가져오게 한 것”임.

## 4. 요약

| 항목 | 내용 |
|------|------|
| **증상** | 개발환경에서만 메인/목록 상품 클릭 시 같은 상품 상세로만 가거나 "상품을 선택해 주세요"만 표시됨. |
| **원인** | (1) 로컬 브라우저 캐시로 예전 JS가 실행되어 링크에 id가 안 붙음. (2) 상세 페이지가 id를 URL에만 의존함. |
| **해결** | 메인에서 클릭 시 `sessionStorage.selectedProductId`에 id 저장 → 상세는 URL 없으면 여기서 읽어서 Firestore로 조회. (같은 데이터 소스 사용) |
| **깃 코드 그대로인데 로컬만 안 됐던 이유** | 로컬에서는 캐시된 예전 HTML/JS가 실행되어, 실제로는 “깃 이전 코드”가 아니라 “예전 캐시”가 동작했기 때문. |
| **앞으로 참고** | “목록 → 상세”처럼 **같은 데이터 소스를 쓰는 흐름**에서는, **URL만 믿지 말고 sessionStorage 등 보조 수단**으로 id를 넘기면, 캐시/링크 오류에 더 견고해짐. |
