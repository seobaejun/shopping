# 상품 상세 페이지 — 개발환경에서만 "한 경로로만 가는" 문제 해결

> 메인에서 상품 클릭 시 상세페이지가 해당 상품이 아니라 "상품을 선택해 주세요" 또는 항상 같은 상품(예: 패리스독)으로만 열리던 문제. **배포(프로덕션)에서는 정상**, **개발(localhost)에서만 발생**했고, sessionStorage 방식으로 해결함.

---

## 1. 어떻게 해결했는지

### 해결 방식: 메인에서 클릭한 상품 ID를 sessionStorage에 넣고, 상세는 URL 없으면 거기서 읽기

- **데이터는 메인과 동일**: 상세페이지도 Firestore에서 같은 상품 문서를 조회함. **다만 상품 ID를 어디서 가져오느냐**만 바꿈.
- **기존**: 상세 진입 시 **URL의 `?id=...` 만** 사용 → 개발환경에서는 이 값이 비어 있거나 잘못 전달되어 문제 발생.
- **변경**:
  1. **메인 페이지**: 상품 카드 클릭 시, 이동하기 **직전에** `sessionStorage.setItem('selectedProductId', id)` 로 클릭한 상품의 문서 ID를 저장한 뒤 `product-detail.html?id=...` 로 이동.
  2. **상세 페이지**: 상품 ID를 정할 때 **URL의 `id`를 먼저 보고**, 없거나 비어 있으면 `sessionStorage.getItem('selectedProductId')` 로 보조해서 사용. 그 ID로 Firestore에서 상품 로드. 로드 성공 시 sessionStorage 항목 제거, 필요 시 `history.replaceState` 로 주소창에 `?id=...` 반영.

이렇게 하면 **URL이 캐시/리다이렉트 등으로 깨져도**, 메인에서 방금 클릭한 상품 ID는 sessionStorage에 남아 있어서, 상세는 **메인과 같은 Firestore**에서 해당 상품만 골라서 보여 줄 수 있음.

### 수정한 파일 요약

| 파일 | 내용 |
|------|------|
| **js/script.js** | 메인 상품 그리드 클릭 시 `sessionStorage.setItem('selectedProductId', id)` 후 `product-detail.html?id=...` 로 이동. (그리드 클릭 시 data-doc-ids에서 id 읽어 이동하는 기존 로직 유지, 그 직전에 sessionStorage 저장 추가.) |
| **js/product-detail.js** | `getProductFromUrl()` 에서 `productId = urlParams.get('id') \|\| sessionStorage.getItem('selectedProductId') \|\| ''` 로 ID 결정. URL에 id 없을 때만 sessionStorage 사용. 로드 성공 시 `sessionStorage.removeItem('selectedProductId')`, 필요 시 `replaceState` 로 URL에 id 반영. |

## 2. 그동안 왜 해결이 안 됐는지 (제가 못 했던 이유)

1. **원인은 캐시인데, 제가 직접 건드릴 수 있는 범위가 아님**  
   - 개발환경에서 **브라우저가 예전에 받아 둔 HTML/JS를 캐시**해서 쓰고 있었음.  
   - 코드를 아무리 바꿔도, **사용자 브라우저의 캐시를 제가 비울 수는 없음**.  
   - 그래서 "수정해도 반영이 안 된다" → 캐시 때문이라고 짚어도, **실제로 캐시를 비우는 건 사용자 동작(강력 새로고침, 사이트 데이터 삭제 등)** 이라, 그걸 제가 대신 실행해 줄 수 없었음.

2. **캐시되는 게 JS만이 아님**  
   - `script.js`만 고쳐도, **index.html이 캐시**돼 있으면 예전 HTML이 로드됨.  
   - 그 예전 HTML이 예전 방식(`<script src="js/script.js?v=101">` 등)으로 스크립트를 부르면, 결국 **예전 JS가 또 캐시에서** 불려 옴.  
   - 그래서 **HTML까지 캐시를 안 하게** 하려면 서버에서 Cache-Control 같은 헤더를 주는 게 필요한데, 그건 `npx serve` 등 서버 설정 영역이라, 코드만으로 완전히 제어하기 어려웠음.

3. **캐시를 우회하려 한 시도들의 한계**  
   - `?v=101`, `?v=Date.now()` 같은 **캐시 버스팅**을 넣었지만, **이미 그 URL로 한 번 받아 둔 사용자**는 그 예전 파일을 계속 쓸 수 있음.  
   - **동적으로 script 로드**하는 방식도, **그걸 담은 index.html 자체가 캐시**돼 있으면 예전 HTML이 실행되어 효과가 없었을 수 있음.  
   - 즉, "한 번은 사용자가 캐시를 비우거나 강력 새로고침을 해줘야 새 코드가 적용된다"는 점을 넘어서지 못함.

4. **진짜 필요한 건 "URL/id 전달에 덜 의존하는 경로"**  
   - 결국 **URL에 id가 안 붙어 오는 환경**에서도, "메인에서 방금 고른 상품"만 정확히 알 수 있으면 됐음.  
   - 그걸 **sessionStorage로 메인→상세에 한 번만 넘기는 방식**으로 바꾸니까, 캐시와 무관하게 동작하게 된 것.

## 3. 깃 이전 코드를 그대로 가져와도 개발환경에서만 왜 안 됐는지

- **같은 코드**가 **배포(프로덕션)** 에서는 잘 동작하고, **개발(localhost)** 에서만 안 됐던 이유는 **캐시 차이** 때문임.

- **배포 환경**  
  - 새로 배포할 때마다 **서버에 새 파일**이 올라가고, 사용자는 **새 URL 또는 새 리소스**를 받음.  
  - 브라우저가 **같은 URL의 예전 파일**을 오래 붙들고 있을 가능성이 상대적으로 낮음.  
  - 그래서 **깃에 있는 그 코드**가 그대로 반영된 상태로 실행됨.

- **개발 환경 (localhost)**  
  - **같은 주소**(`http://localhost:5000/`, `js/script.js` 등)를 **반복해서** 연다.  
  - 브라우저는 "이미 이 URL로 받아 둔 게 있다"고 보고 **캐시에서** HTML/JS를 씀.  
  - 그래서 **디스크/깃에는 최신 코드**가 있어도, **실제로 실행되는 건 예전에 캐시된 파일**이라,  
    - 메인에서 만든 링크에 `id`가 안 붙거나,  
    - 상세로 갈 때 `?id=...` 가 빠지는 현상이 **개발에서만** 계속됐던 것.

- **정리**  
  - "깃 이전 코드를 그대로 가져와도" **파일 내용**은 프로덕션과 같지만,  
  - **개발에서는 그 코드가 아닌, 예전에 캐시된 버전이 실행**되고 있었기 때문에,  
  - **같은 코드인데도 개발환경에서만 안 됐던 것**으로 보면 됨.

---

## 4. 앞으로 비슷한 문제가 나올 때

- **증상**: 배포에서는 정상인데, 로컬(localhost)에서만 링크/상세가 이상할 때.  
- **우선**:  
  1. **로컬 캐시 비우기** (강력 새로고침, 사이트 데이터 삭제, 시크릿 창 등)로 **새 코드가 실제로 로드되는지** 확인.  
  2. 그래도 안 되면, **URL/id 전달에 덜 의존**하도록 **sessionStorage(또는 동일 출처에서 쓸 수 있는 다른 수단)** 로 "어디서 왔는지 / 어떤 상품인지"를 넘기는 방식 검토.  
- **참고**: 로컬 캐시 비우는 방법은 `SERVER_TROUBLESHOOTING.md` 또는 `wishlist-cart-pc-difference.md` 의 "배포 후 캐시 무효화" 절과 맞춰 두면 됨.
