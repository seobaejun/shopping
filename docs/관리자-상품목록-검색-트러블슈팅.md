# 관리자 상품목록 검색 — 왜 안 됐는지, 해결 방법

> 다음에 관리자 검색(또는 비슷한 기능)이 안 될 때 이 문서를 참고하세요.

---

## 1. 최종 해결 방식

**상품목록 검색은 `admin.js`에서만 처리합니다.**

- **구매요청 검색**과 동일한 패턴으로 구현함.
- `admin.js`에 **`applyProductListSearch()`** 함수 추가.
- 폼 전송(submit) 시 **`applyProductListSearch()`** 만 호출.
- **product-list.js에 전혀 의존하지 않음** (검색 버튼/폼 → product-list.js의 `searchProducts` 호출 제거).

**데이터 흐름:**

1. `window.firebaseAdmin.productService.getProducts()` 로 상품 목록 조회 (다른 관리자 기능과 동일).
2. 상품명·카테고리·상태는 **클라이언트에서 필터링**.
3. 결과만 **`productSearchResultsBody`** / **`searchProductCount`** 에 직접 렌더링.

---

## 2. 그동안 못 했던 이유 (원인 정리)

### 2.1 이벤트/바인딩 문제

- **증상:** 검색 버튼을 눌러도 콘솔에 아무 로그도 안 뜸.
- **원인 후보:**
  - 인라인 `onclick`이 CSP 등으로 실행되지 않음.
  - `product-list.js` 로드 순서/시점 때문에 `window.searchProducts`가 아직 없거나, 다른 스크립트가 덮어씀.
  - document / #product-list 위임 리스너가 등록되기 전에 사용자가 클릭하거나, 다른 요소가 클릭을 가로챔.
- **시도했던 것:** 인라인 onclick 추가/제거, document submit 위임, #product-list 캡처 리스너, 폼 submit으로 전환 등 → **근본적으로 “어디서 검색 로직을 실행할지”가 불안정했음.**

### 2.2 검색 로직이 product-list.js에만 있던 문제

- **증상:** 폼 전송은 감지되는데 “검색 중…”에서 멈춤.
- **원인:**
  - 검색 실행을 **product-list.js의 `searchProducts()`에만** 의존함.
  - 그 함수는 **`window.allProductsData` / `allProductsData`** 를 사용하는데, 이 값은 **상품목록 페이지 로드 시 `loadAllProducts()`가 끝난 뒤에만** 채워짐.
  - 데이터가 비어 있으면 `searchProducts()` 안에서 **다시 `loadAllProducts()` → Firestore `getProducts()`** 를 호출하고 **응답이 올 때까지 대기**.
- **결과:** Firebase/네트워크가 느리거나 응답이 없으면 **무한 대기 → “검색 중…”에서 멈춤**.  
  (타임아웃을 넣어도, 근본적으로 “검색 시 Firebase 대기”에 의존하는 구조가 문제.)

### 2.3 admin.js와 product-list.js가 서로 다른 역할을 하던 문제

- **admin.js** 에도 **`searchProducts`** 와 **`renderProductTable()`** 이 있음.
- **`renderProductTable()`** 은 **`productListBody`** / **`productCount`** 에 그리는데,  
  **상품목록 페이지**에서는 **`productListPagination`** 이 있으면 **“이미 product-list.js가 전체 목록을 그렸다”고 보고 early return** 함.
- 그래서 admin.js의 `searchProducts()` → `renderProductTable(filtered)` 를 호출해도 **아무 것도 그려지지 않음** (검색 결과가 표시되지 않음).
- 한편 **검색 결과를 보여줘야 할 영역**은 **`productSearchResultsBody`** / **`searchProductCount`** 인데, admin.js는 이 영역을 채우는 로직이 없었음.

### 2.4 정리: 왜 수천 번 시도해도 안 됐는가

| 시도한 방향 | 한계 |
|------------|------|
| 버튼/폼 이벤트만 고침 | 실행 주체가 product-list.js / admin.js 사이에서 불명확했고, product-list.js 의존 시 로드·데이터 타이밍에 취약. |
| product-list.js에서 Firebase 대기 제거 | “데이터 없으면 검색 안 함”으로 바꿔도, **검색 실행 자체**가 product-list.js에 묶여 있어서 이벤트/로드 문제는 그대로. |
| 타임아웃 추가 | “멈춤”만 완화할 뿐, **검색 결과를 어디에, 어떤 방식으로 그릴지**를 admin.js와 통일하지 않음. |

**결국:**  
“검색 실행 + 데이터 조회 + 결과 표시”를 **다른 검색(구매요청 등)과 동일하게 admin.js 한 곳에서 처리**하지 않아서, 이벤트·데이터·렌더링이 서로 엇갈린 상태가 오래 지속됨.

## 3. 해결의 핵심 (다음에 참고할 때)

1. **관리자에서 “검색”이 안 될 때**
   - **같은 페이지의 다른 검색(구매요청, 정산, 배송 등)은 어떻게 데이터를 가져오고, 어느 테이블에 그리는지** 먼저 확인.
   - 그 검색은 **어느 스크립트(보통 admin.js)에서** 데이터 조회 + 필터 + 테이블 렌더링을 하는지 파악.

2. **“검색 버튼이 반응 없다”**
   - 인라인/위임/폼 submit 등 **이벤트만 바꾸지 말고**,  
     **“검색 로직을 실행하는 쪽”을 이미 잘 동작하는 검색과 동일한 스크립트·같은 패턴**으로 옮기기.

3. **“검색 중에서 멈춘다”**
   - 검색 시 **추가로 Firebase/API를 호출하고 그 응답을 기다리는지** 확인.
   - **이미 다른 탭/페이지에서 불러온 데이터**로 필터만 하면 되는지,  
     아니면 **해당 관리자 페이지가 쓰는 데이터 조회 방식(getProducts 등)**을 그대로 쓰는지 정리.

4. **표시 영역이 다르다**
   - “전체 목록용 테이블”과 “검색 결과용 테이블”의 **id**(예: `productListBody` vs `productSearchResultsBody`)를 구분.
   - **검색 결과는 검색 결과용 tbody/count에만** 그리도록 하고,  
     다른 스크립트의 `renderProductTable` 같은 함수가 **검색 결과 영역을 건드리지 않고 early return** 하지 않도록,  
     **검색 전용 렌더 함수**(예: `applyProductListSearch`)를 두는 것이 안전함.

---

## 4. 관련 코드 위치 (참고)

- **검색 실행:** `admin.js` — `document.addEventListener('submit', ...)` (id `productSearchForm`), 내부에서 `applyProductListSearch()` 호출.
- **검색 로직·렌더:** `admin.js` — `applyProductListSearch()` (Firebase getProducts → 필터 → `productSearchResultsBody` / `searchProductCount` 에 직접 렌더).
- **비교용 패턴:** `admin.js` — `applyPurchaseRequestSearch()` (구매요청 검색, 동일한 “데이터 가져오기 → 필터 → 검색 결과 테이블만 갱신” 방식).
