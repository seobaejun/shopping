# 주소 표시 문제 디버깅 가이드

## 문제 상황
- 마이페이지에서는 주소가 표시되지만, product-detail 페이지에서는 "등록된 주소가 없습니다" 표시
- 사용자는 분명히 주소 데이터가 있다고 주장

## 해결 과정에서 범한 실수들

### ❌ 잘못된 접근 방법들

1. **CSS 문제라고 착각**
   - 표시 문제를 스타일링 문제로 오해
   - 실제로는 데이터 자체를 못 가져오는 문제였음

2. **Firebase 초기화 문제라고 착각**
   - Firebase가 늦게 로드되어서 데이터를 못 가져온다고 생각
   - 실제로는 데이터 구조 이해 부족

3. **localStorage 문제라고 착각**
   - 로그인 정보가 localStorage에 제대로 저장되지 않았다고 생각
   - 실제로는 Firestore 데이터 읽기 방식의 차이

### ✅ 올바른 디버깅 순서

## 1단계: 작동하는 코드부터 분석하기

**핵심 원칙**: 같은 데이터를 사용하는데 한 곳에서는 되고 한 곳에서는 안 된다면, **작동하는 곳의 코드를 먼저 분석**해야 함

```javascript
// 마이페이지에서 주소를 어떻게 가져오는지 확인
// mypage.js -> fillProfileForm() 함수
function fillProfileForm(member) {
    set('profilePostcode', member.postcode);      // ✅ 최상위 필드 사용
    set('profileAddress', member.address);        // ✅ 최상위 필드 사용  
    set('profileDetailAddress', member.detailAddress); // ✅ 최상위 필드 사용
}
```

## 2단계: 데이터 구조 정확히 파악하기

```javascript
// Firestore members 컬렉션의 실제 데이터 구조
{
  // 최상위 필드 (마이페이지가 사용)
  postcode: "549",
  address: "전북특별자치도 전주시 완산구 전주천서로 445", 
  detailAddress: "110 동 601호",
  name: "김병욱",
  phone: "010-4842-2304",
  
  // addresses 배열 (배송지 관리에서 사용, 비어있을 수 있음)
  addresses: [] // ← 이게 비어있어서 문제 발생
}
```

## 3단계: 문제 코드와 작동 코드 비교

### ❌ 문제가 있던 product-detail.js
```javascript
// addresses 배열에서만 주소를 찾으려고 시도
const defaultAddress = addresses.find(addr => addr.isDefault === true) || {};

member = {
    // addresses 배열이 비어있으면 빈 문자열이 됨
    postcode: defaultAddress.postcode || '',  // '' (빈 문자열)
    address: defaultAddress.address || '',    // '' (빈 문자열)
    detailAddress: defaultAddress.detailAddress || '' // '' (빈 문자열)
};
```

### ✅ 수정된 product-detail.js (마이페이지 방식 적용)
```javascript
member = {
    // 최상위 필드 우선, addresses 배열은 보조
    postcode: memberData.postcode || defaultAddress.postcode || '',
    address: memberData.address || defaultAddress.address || '',
    detailAddress: memberData.detailAddress || defaultAddress.detailAddress || ''
};
```

## 4단계: 사용자 제공 디버깅 정보 꼼꼼히 분석

사용자가 제공한 콘솔 로그:
```
문서 ID: sprince1004 데이터: { 
  name: '', phone: '', postcode: '', address: '', detailAddress: '', 
  addresses: [] ... 
  userId: 'sprince1004', email: 'sprince1004@naver.com' 
}
```

**실수**: 이 로그를 보고 "데이터가 없다"고 판단
**실제**: 이것은 `sprince1004` 사용자의 데이터이고, 문제를 제기한 사용자는 `김병욱`이었음

## 핵심 교훈

### 1. **데이터 흐름 우선 분석**
- UI 문제가 아니라 데이터 문제일 가능성을 먼저 고려
- 같은 데이터를 사용하는 다른 페이지와 비교 분석

### 2. **작동하는 코드를 참고 기준으로 사용**
- 마이페이지가 작동한다면, 마이페이지의 방식을 따라야 함
- 새로운 방식을 시도하기보다는 검증된 방식 사용

### 3. **데이터 구조 완전 이해**
```javascript
// Firestore 데이터 구조 이해
{
  // 개인정보 (최상위 필드)
  name: "김병욱",
  phone: "010-4842-2304", 
  postcode: "549",           // ← 마이페이지는 이걸 사용
  address: "전북특별자치도...", // ← 마이페이지는 이걸 사용
  detailAddress: "110 동 601호", // ← 마이페이지는 이걸 사용
  
  // 배송지 관리 (배열)
  addresses: [               // ← product-detail.js가 이것만 보고 있었음
    {
      recipientName: "김병욱",
      phone: "010-4842-2304",
      postcode: "549",
      address: "전북특별자치도...",
      detailAddress: "110 동 601호",
      isDefault: true
    }
  ]
}
```

### 4. **디버깅 정보 정확히 해석**
- 사용자가 제공한 정보와 실제 문제 상황이 일치하는지 확인
- 콘솔 로그의 사용자 ID와 문제를 제기한 사용자가 같은지 확인

### 5. **단계별 접근**
1. 작동하는 코드 분석 (마이페이지)
2. 문제 코드와 비교 (product-detail.js)
3. 데이터 구조 파악 (Firestore)
4. 차이점 식별 및 수정
5. 테스트 및 검증

## 다음에 같은 문제가 발생하면

1. **"마이페이지에서는 되는데 여기서는 안 된다"** → 마이페이지 코드부터 분석
2. **"데이터가 있는데 표시가 안 된다"** → 데이터 가져오는 방식 비교
3. **Firebase/localStorage 의심하기 전에** → 데이터 구조와 접근 방식 확인
4. **CSS/UI 문제로 착각하지 말고** → 데이터 흐름부터 확인

## 결론

**근본 원인**: product-detail.js가 `addresses` 배열에서만 주소를 찾고, 마이페이지가 사용하는 최상위 필드(`postcode`, `address`, `detailAddress`)를 무시하고 있었음

**해결 방법**: 마이페이지와 동일하게 최상위 필드를 우선적으로 사용하도록 수정

**교훈**: 작동하는 코드(마이페이지)를 먼저 분석했어야 했음
